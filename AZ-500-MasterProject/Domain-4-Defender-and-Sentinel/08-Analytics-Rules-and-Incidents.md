# Microsoft Sentinel Analytics Rules and Incidents

> 📌 AZ-500 Exam Objective: Configure analytics rules and manage incidents in Microsoft Sentinel
> 🏷️ Domain: 4 — Defender for Cloud and Sentinel | Weight: 30–35%

***

## What Is This?

Analytics rules are the detection engine of Microsoft Sentinel. They are queries that run automatically on a schedule — or in near real-time — against your log data. When a rule finds a match, it fires an alert. Multiple related alerts get grouped into one incident. Incidents are what your SOC team investigates and resolves.

***

## Why Does This Exist?

Raw log data does not tell you when something is wrong. Analytics rules are the logic layer that turns raw logs into actionable alerts. Without them, Sentinel collects data but never flags anything. Analytics rules let you encode attack patterns — like failed logins followed by a successful login, or a new admin account being created at 3 AM — and get notified automatically every time those patterns appear.

***

## Who Actually Needs This? (Real-World Context)

A cybersecurity team at a law firm called LexShield needs to detect when anyone tries to access confidential client files outside business hours. They create a scheduled analytics rule that queries the `OfficeActivity` table for SharePoint file access between 8 PM and 6 AM. They map the rule to the MITRE ATT&CK tactic "Collection" and set the severity to High. The first week, the rule fires on a contractor who was legitimately working late. The analyst closes the incident with notes. The second week, the rule fires on an account that was already deactivated — a sign of compromised credentials. The analyst escalates immediately.

***

## How It Works (Pure Concept, No Portal)

**Analytics Rule Types**

Sentinel has six types of analytics rules:

**1. Scheduled Rules**
- The most common type.
- You write a KQL query that runs on a defined schedule (e.g., every 5 minutes, every 1 hour).
- If the query returns results, an alert is created.
- You define: query, frequency (how often to run), lookback period (how far back to search), alert threshold (minimum results to trigger), entity mapping, and MITRE tactic.

**2. Near Real-Time (NRT) Rules**
- Runs approximately every 1 minute — nearly in real time.
- Uses a simpler query structure than scheduled rules.
- Best for high-priority detections where speed matters (e.g., a privileged account signing in from a new country).
- Limited to simpler KQL (no complex joins or long lookback periods).

**3. Microsoft Security Rules**
- Pre-built rules that automatically create Sentinel incidents from alerts generated by other Microsoft security products.
- Examples: create a Sentinel incident whenever Microsoft Defender for Endpoint generates a High alert, or whenever Defender for Cloud fires a critical alert.
- No KQL writing needed — you just configure severity filters.

**4. Fusion Rules**
- Machine learning rules that correlate signals across multiple data sources.
- Detects multi-stage attacks that no single rule would catch.
- Example: Fusion might detect a combination of impossible travel sign-in + mass file download + new inbox forwarding rule as a single coordinated attack scenario.
- Fusion is enabled by default and cannot be edited.

**5. Anomaly Rules**
- Machine learning rules that learn your environment's baseline behavior.
- Fire alerts when behavior deviates significantly from the baseline.
- Examples: unusual volume of data exported, user accessing resources they have never accessed before.
- Uses UEBA (User and Entity Behavior Analytics) under the hood.

**6. Threat Intelligence Rules**
- Automatically match log data against threat intelligence indicators (IOCs) stored in the `ThreatIntelligenceIndicator` table.
- When a known malicious IP, domain, or file hash appears in your logs, an alert fires.
- Requires a threat intelligence data connector (e.g., MISP, Threat Intelligence Platform connector).

**Rule Configuration — Key Fields**

When building a Scheduled rule, you configure:

| Field | What It Controls |
|---|---|
| Query | The KQL logic that defines what to detect |
| Query frequency | How often the rule runs (every 5 min, 1 hour, 1 day) |
| Lookback period | How far back in time the query searches (must be >= frequency) |
| Alert threshold | Minimum number of results needed to create an alert (default: > 0) |
| Entity mapping | Tells Sentinel which query columns represent accounts, IPs, hosts, URLs, files |
| Alert grouping | Controls whether multiple alerts from the same rule get merged into one incident |
| MITRE ATT&CK tactics | Tags the alert with the MITRE technique (e.g., T1078 - Valid Accounts, TA0001 - Initial Access) |
| Severity | High, Medium, Low, or Informational |

**Alerts vs Incidents — A Critical Distinction**

This is one of the most commonly confused topics on the AZ-500 exam.

- **Alert** = a single detection event from one analytics rule firing once.
- **Incident** = a case in Sentinel that groups one or more related alerts together for investigation.

By default, each alert creates its own incident. But with alert grouping, you can configure rules to group multiple alerts into one incident if they share the same entity (e.g., the same user account or the same IP address) within a time window. This prevents SOC analysts from being overwhelmed by dozens of separate incidents for what is really one ongoing attack.

**Incident Workflow**

1. Incident is created (automatically by analytics rule or manually).
2. Analyst **assigns** the incident to a team member.
3. Analyst **investigates** — views the investigation graph, linked alerts, entities, and related logs.
4. Analyst adds **comments** and **bookmarks** to key evidence.
5. Analyst takes **action** — runs a playbook, blocks an IP, disables a user.
6. Analyst **closes** the incident with a classification: True Positive, False Positive, Benign Positive, or Undetermined.

**Investigation Graph**
The investigation graph is a visual map of an incident. It shows the entities involved (users, IPs, hosts, resources) and the relationships between them. You can explore the graph by clicking any entity to see what other alerts, activities, or connections are associated with it.

## 2026 Portal Walkthrough — Step by Step

**Create a Scheduled Analytics Rule:**
1. In Microsoft Sentinel, click **Analytics** in the left menu.
2. Click **Create** → **Scheduled query rule**.
3. On the **General** tab:
   - Name: `Failed Sign-ins Brute Force Detection`
   - Description: Detects 10 or more failed sign-ins from one IP in 1 hour.
   - Severity: High
   - MITRE ATT&CK: Select **Credential Access** → **Brute Force (T1110)**
   - Status: Enabled
4. On the **Set rule logic** tab:
   - Paste your KQL query.
   - Set **Query scheduling**: Run query every **1 hour**.
   - Set **Lookup data from the last**: **1 hour**.
   - Set **Alert threshold**: Generate alert when number of results is **greater than 0**.
5. On the **Incident settings** tab:
   - Enable incident creation: **On**.
   - Alert grouping: **Group alerts into a single incident if matching entity is IPAddress within 1 hour**.
6. On the **Entity mapping** tab:
   - Map `IPAddress` column to entity type **IP**.
   - Map `UserPrincipalName` column to entity type **Account**.
7. Click **Review and create** → **Save**.

**Configure Entity Mapping:**
1. In the rule wizard, go to the **Entity mapping** section.
2. Click **Add new entity**.
3. Select entity type (Account, IP, Host, URL, File, Process).
4. Map the entity identifier to the matching column in your KQL query output.
5. Entity mapping enables the investigation graph and links incidents to entity pages (like a user's full activity timeline).

**View an Incident:**
1. Click **Incidents** in the Sentinel left menu.
2. Filter by **Status: Active** and **Severity: High**.
3. Click an incident to open the details pane.
4. Click **View full details** to open the incident page.
5. Click **Investigate** to open the investigation graph.
6. Click any entity node to see related activity and alerts.
7. Assign the incident to yourself by clicking **Assign to me**.
8. When done, click **Change status** → **Closed** and select a classification.

## Key Settings Explained

| Setting | What It Does | When to Use |
|---|---|---|
| Query frequency | How often the rule runs | Set to match the urgency — 5 min for critical rules, 1 hour for lower priority |
| Lookback period | How far back the query searches each time it runs | Must be at least equal to frequency; set longer for rules that need context across multiple events |
| Alert threshold | Minimum results to trigger an alert | Set to > 0 for simple rules; set higher to reduce false positives on noisy data |
| Entity mapping | Links query output columns to Sentinel entity types | Always map — required for investigation graph and UEBA correlation |
| Alert grouping | Merges multiple alerts into one incident based on shared entities | Enable to reduce noise when one attack generates many alerts |
| MITRE tactic mapping | Tags alerts with attack framework categories | Helps prioritize and understand the attack stage |
| Suppression | Pauses the rule after it fires for a set period | Use to prevent alert storms when you know a noisy event is expected |

## Comparison Table (if applicable)

| Rule Type | Detection Method | Best For | Requires KQL? |
|---|---|---|---|
| Scheduled | KQL query on a schedule | Custom detections, complex logic | Yes |
| NRT (Near Real-Time) | KQL query every ~1 minute | High-priority, low-latency detection | Yes (simpler) |

## What Happens If You Skip This?

Without analytics rules, Sentinel collects logs but never fires alerts. You have a data lake with no filters. Attacks that match well-known patterns — password spraying, lateral movement, privilege escalation — happen silently and go completely undetected. Your SOC team spends time manually searching logs reactively after a breach is already discovered, instead of detecting threats in progress.

## AZ-500 Exam Section

- **Trigger Words:** "analytics rule", "scheduled rule", "NRT rule", "incident", "alert grouping", "MITRE", "entity mapping", "Fusion", "alert threshold", "lookback period"
- **Common Traps:**
  - Thinking each alert creates a new incident — alert grouping combines related alerts into one incident based on shared entity and time window.
  - Thinking NRT rules work exactly like scheduled rules — NRT rules run every ~1 minute but support only simpler KQL. You cannot use complex joins with long lookbacks in NRT rules.
  - Thinking Fusion rules can be customized — Fusion rules are built-in ML detections that cannot be edited.
  - Thinking alert severity is the same as incident severity — incident severity is set by Sentinel based on the highest-severity alert in the incident (or by automation rules that override it).
- **How to Tell Apart:**
  - Scheduled rule = you write the KQL, runs on a timer
  - NRT rule = you write the KQL, runs every ~1 minute
  - Microsoft Security rule = auto-creates incidents from Defender product alerts, no KQL
  - Fusion rule = Microsoft ML, cannot edit, correlates across data sources
- **Likely Question Formats:**
  - "You need Sentinel to automatically create an incident whenever Microsoft Defender for Endpoint generates a High-severity alert. Which rule type do you use?" → Microsoft Security rule.
  - "A SOC analyst sees 50 separate incidents from the same brute force attack. How can you reduce this to one incident?" → Enable alert grouping on the analytics rule, grouping by IPAddress entity.
  - "Which analytics rule type requires the lowest latency — running every ~1 minute?" → Near Real-Time (NRT) rule.
- **Memorization Tip:** Think of analytics rules as **fishing nets**. Scheduled rules are large nets you cast every hour. NRT rules are small fast nets you cast every minute. Microsoft Security rules are nets another tool already cast — you just pick up what they caught. Fusion is an AI net that catches what all the other nets missed.

## Pocket Summary (5-Year Refresh)

- Analytics rules are the detection engine in Sentinel — they run KQL queries and create alerts when threats are found.
- Six rule types: Scheduled, NRT, Microsoft Security, Fusion, Anomaly, and Threat Intelligence — each serves a different detection need.
- Alerts are individual detections; incidents are the investigation cases that group related alerts together.
- Alert grouping is critical for reducing noise — it combines multiple alerts into one incident when they share the same entity (user, IP, host) within a time window.
- Always configure entity mapping — it powers the investigation graph and links incidents to full user/host activity timelines.
